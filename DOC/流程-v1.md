## 1. 合并文档 A：普拉奇克8情绪玩法规格（v1）

# 普拉奇克8情绪玩法规格（v1）

## 1. 目标

将现有情绪玩法改为：

1. 从角色池中按“玩家勾选允许出现角色”选择角色（严格模式下不再随机补足）。
2. 调用大模型分析角色情绪，输出 8 基础情绪权重。
3. 玩家每轮猜测 3 种情绪（不设置猜错惩罚，不阻断流程）。
4. 根据玩家猜测的 3 种情绪生成调酒目标。
5. 若命中角色权重前 3 的情绪越多，则获得更高小费与信任值。

---

## 2. 情绪集合与标识

固定 8 种基础情绪（内部 ID）：

- joy（Joy / 喜悦）
- trust（Trust / 信任）
- fear（Fear / 恐惧）
- surprise（Surprise / 惊讶）
- sadness（Sadness / 悲伤）
- disgust（Disgust / 厌恶）
- anger（Anger / 愤怒）
- anticipation（Anticipation / 期待）

约束：

1. 游戏内所有情绪推理、校验、存档均以以上 8 个 ID 为唯一合法值。
2. 模型返回非合法值时，必须过滤并走降级策略。

---

## 3. 角色抽取规则

数据源：polyu-storyworld 子模块中的 characters 目录。

抽取规则：

1. 仅纳入扩展名 .yaml/.yml 的文件。
2. 当日顾客优先从 activeCharacterIds 对应的角色文件中选择。
3. 若 activeCharacterIds 为空或首位顾客生成失败，则提示返回配置页重新添加角色，不走随机补足。
4. 若文件解析失败，则按失败提示处理，不再使用本地降级角色模板。

建议额外字段抽取（用于情绪分析）：

- name
- profile / description
- background / biography
- personality / traits
- dialogue_style（若存在）

---

## 4. AI 情绪分析输出协议

模型必须返回 JSON（禁止自由文本），建议结构：

```json
{
  "weights": {
    "joy": 0.12,
    "trust": 0.18,
    "fear": 0.10,
    "surprise": 0.08,
    "sadness": 0.14,
    "disgust": 0.07,
    "anger": 0.11,
    "anticipation": 0.20
  },
  "top3": ["anticipation", "trust", "sadness"],
  "confidence": 0.82,
  "rationale": [
    "...",
    "..."
  ]
}
```

后处理硬规则：

1. 必须包含 8 个键；缺失键补 0。
2. 对所有权重进行 clamp 到 [0,1]。
3. 归一化：sum(weights) = 1。
4. top3 由系统根据排序重算，不信任模型原值。
5. 置信度缺失时默认 0.6。

---

## 5. 玩家猜测规则

1. 每轮玩家必须选择 3 种情绪（不足 3 种不可确认）。
2. 猜测不会失败，不中断调酒流程。
3. 不设置猜错扣分（不降信任、不罚钱）。

命中定义：

- 令 guessed3 为玩家选择集合。
- 令 top3 为系统计算出的真实前三情绪集合。
- 命中数 h = |guessed3 ∩ top3|，h 取值 0~3。
- 令 surfaceSet 为当前可观察到的表象情绪集合（建议为 1~2 个）。
- 表象命中数 hs = |guessed3 ∩ surfaceSet|，hs 取值 0~3。

---

## 6. 调酒目标生成（由猜测驱动）

### 6.1 八情绪口感原型（初版）

每个情绪映射一个三维原型向量（thickness, sweetness, strength）：

- joy: (1, 2, 1)
- trust: (2, 1, 1)
- fear: (0, 0, 2)
- surprise: (1, 1, 2)
- sadness: (2, 0, 1)
- disgust: (1, 0, 2)
- anger: (0, 0, 3)
- anticipation: (1, 2, 2)

### 6.2 合成公式

玩家猜测 3 种情绪，默认等权合成目标向量：

T = (E1 + E2 + E3) / 3

再将目标向量转换为判定区间（建议每维容差 ±0.5）。

说明：

1. 后续可升级为非等权（玩家可指定主情绪权重）。
2. 若目标在当前材料池无解，执行“逐步放宽”策略（与现有 solvable target 机制一致）。

---

## 7. 奖励与信任规则

按照命中数 h 进行奖励分档：

- h=3：tipMultiplier = 1.80，trustGain = 0.18
- h=2：tipMultiplier = 1.40，trustGain = 0.10
- h=1：tipMultiplier = 1.15，trustGain = 0.05
- h=0：tipMultiplier = 1.00，trustGain = 0.02

表象命中额外信任（小额补贴，不影响小费倍率）：

- 每命中 1 个表象情绪，额外增加 extraTrustPerHit = 0.015
- 单杯额外信任上限 extraTrustCap = 0.03

即：

- extraTrust = min(0.015 × hs, 0.03)
- finalTrustGain = trustGain + extraTrust

结算方式：

1. 小费 = 基础小费 × tipMultiplier × 调酒质量系数。
2. 信任增加 = finalTrustGain（并可叠加调酒质量微调）。
3. 禁止因猜测错误产生负向惩罚。
4. 表象命中只提供小额正反馈，底层命中仍是主要收益来源。

---

## 8. 平局与边界处理

1. 权重并列时，使用固定优先级打破平局：
   joy > trust > anticipation > surprise > sadness > fear > anger > disgust
2. 若模型返回异常或解析失败：使用均匀分布（每项 0.125）。
3. 若玩家未满 3 选：确认按钮禁用。
4. 若角色资料极短：仍可分析，但 confidence 上限建议限制为 0.65。

## 2. 合并文档 B：MCP角色调用与ComfyUI像素生图计划（v1）

# MCP角色调用与ComfyUI像素生图计划（v1）

## 1. 目标与范围

本计划用于在当前项目中新增一套 MCP 工具层，实现以下核心能力：

1. 用户输入角色名（如 `5738g`）后，系统能精准读取 `polyu-storyworld` 角色 YAML。
2. 自动提取角色设定并进行 8 维情绪分析（Plutchik 8 基础情绪）。
3. 基于角色剧情设定与情绪结果，生成可用于玩家对话的上下文。
4. 按用户输入角色名调用 ComfyUI 生成像素风角色图（异步任务）。

非目标（v1 暂不覆盖）：

1. 完整替换现有全部顾客生成逻辑。
2. 多风格并行生图编排（先实现像素风单工作流）。
3. 复杂权限系统（先做本地开发可用）。

---

## 2. 参考形态（对齐 ltx2-comfy-v915 的思路）

参考仓库的关键思想不是“照搬代码”，而是“工具化 + 工作流化”：

1. 将外部 AI 能力封装为可调用工具，而不是散落在业务逻辑里。
2. 将 ComfyUI 调用拆分为「提交任务 -> 查询状态 -> 获取结果」的标准流程。
3. 将提示词模板、工作流 JSON、角色数据解析逻辑模块化管理。
4. 保留可替换性：后续可替换模型、工作流、提示词而不重写主流程。

---

## 3. 总体架构

建议新增一个独立目录：`mcp-server/`（Node.js 实现）。

数据流：

1. 前端/游戏层输入角色名。
2. MCP 工具 `character.get_by_name` 读取并解析 YAML。
3. MCP 工具 `emotion.analyze_character` 输出标准化 8 维权重与 top3。
4. MCP 工具 `dialogue.build_context` 组装剧情+情绪+当前回合上下文。
5. （可选）MCP 工具 `image.generate_pixel_portrait` 提交 ComfyUI 任务。
6. 前端轮询 `image.get_task` 获取状态与最终图片地址。

---

## 4. MCP 工具清单（v1）

### 4.1 角色工具

1. `character.get_by_name`
- 输入：`{ name: string }`
- 行为：
  - 在角色目录中查找 `name.yml` / `name.yaml`。
  - 解析 YAML，提取字段：`name/profile/description/background/biography/personality/traits/dialogue_style`。
  - 返回标准角色对象。
- 异常：文件不存在、解析失败。

2. `character.search`
- 输入：`{ keyword?: string, limit?: number }`
- 行为：返回可选角色列表（用于输入联想与调试）。

### 4.2 情绪工具

1. `emotion.analyze_character`
- 输入：`{ character: CharacterPayload }` 或 `{ name: string }`
- 输出（标准协议）：
  - `weights`（8 维）
  - `top3`（系统重算）
  - `confidence`
  - `rationale`
- 后处理硬规则：
  - 仅允许 8 个合法情绪 ID：
    `joy trust fear surprise sadness disgust anger anticipation`
  - 缺失补 0、clamp 到 [0,1]、归一化到 sum=1。
  - 置信度缺失默认 `0.6`。
  - 解析失败回退均匀分布 `0.125`。

### 4.3 对话上下文工具

1. `dialogue.build_context`
- 输入：
  - `character`
  - `emotionAnalysis`
  - `gameState`（可选：日期、信任值、最近事件）
- 输出：
  - `systemPromptPatch`
  - `npcPersonaSummary`
  - `sceneHooks`（可用于回合对话）

### 4.4 ComfyUI 生图工具

1. `image.generate_pixel_portrait`
- 输入：
  - `name`
  - `character`
  - `style`（默认 `pixel-portrait-v1`）
- 行为：
  - 根据角色设定生成正负提示词。
  - 加载预设 workflow JSON。
  - 调用 ComfyUI `/prompt` 提交任务。
- 输出：`{ taskId, status: "queued" }`

2. `image.get_task`
- 输入：`{ taskId }`
- 输出：`queued/running/succeeded/failed` + 图片路径/错误信息。

## 8. 关键技术决策

1. 角色名安全校验
- 仅允许：`[A-Za-z0-9_-]`。
- 禁止路径穿越（如 `../`）。

2. 情绪模型容错
- 不信任模型原始 `top3`，系统重算。
- 返回脏数据时强制清洗，不影响流程。

3. ComfyUI 调用策略
- 默认异步，不阻塞主线程。
- 超时可配置（例如 120s）。
- 支持失败重试 1 次（可选）。

4. 可观测性
- 记录 tool 调用耗时、失败原因、命中缓存率。

## 9. 风险与对策

1. 风险：角色 YAML 字段不统一
- 对策：做字段映射表与多候选提取（profile/description 等）。

2. 风险：ComfyUI 工作流节点 ID 不稳定
- 对策：固定 workflow 模板并加启动自检。

3. 风险：模型输出不稳定

4. 风险：前端等待时间过长
- 对策：异步任务 + 占位图 + 进度提示。
